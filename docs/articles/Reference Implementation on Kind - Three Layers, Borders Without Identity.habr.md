# Референс-реализация на Kind: три слоя и границы без identity

В первой части был нарисован архитектурный каркас self-hosted enterprise AI harness. Здесь он перестаёт быть схемой и превращается в реальный запуск: три из четырёх слоёв поднимаются на локальном Kind-кластере, с двумя изолированными тенантами и end-to-end маршрутом через весь стек.

Четвёртый слой в этой реализации отсутствует намеренно. Это не упущение, а сознательное решение, чтобы не перегружать материал. **Identity, Policy & Audit** логичней разобрать отдельно, иначе текст разрастется и потеряет фокус.

> **Оговорка о контексте.** Описанный здесь запуск — промежуточный срез: три слоя и границы, identity намеренно вынесен в следующую часть. В полном проекте к моменту публикации Identity на границе уже реализован (identity-провайдер, SSO, валидация токенов на gateway), а эта статья фиксирует этап, на котором система уже разделяла тенантов, но ещё не различала пользователей внутри тенанта.

## Контекст

Основная мысль этой реализации простая: проблема self-hosted enterprise AI harness обычно не в выборе компонентов, а в том, где проходят границы между ними.

В этой статье поднимаются первые три слоя:

- Input.
- Agent Loop(ReAct).
- Execution.

Слой Identity, Policy & Audit здесь сознательно отсутствует. Вместе с ним отсутствуют и связанные механизмы: AgentGateway, JWT-валидация, CEL-политики, OTEL-аудит, Vault как секрет-брокер и HITL.

## Что именно проверяется

Эта реализация отвечает на очень конкретный вопрос: можно ли безопасно разделять тенанты без полноценной identity-модели.

Ответ да, но только при условии, что границы проведены на нескольких уровнях сразу:

- Kubernetes namespaces.
- NetworkPolicy.
- PostgreSQL Row-Level Security.
- Tenant-scoped secrets.
- Ограничение cross-tenant вызовов на уровне kagent.

Это не замена identity. Это промежуточное состояние, в котором система уже умеет разграничивать тенанты друг от друга, но ещё не умеет различать роли внутри тенантов.

## Границы без идентичности

Нужно различать две разные границы.

Первая — это **граница идентичности**. Она отвечает на вопрос, кто именно делает действие и по какому праву.

Вторая — **инфраструктурная граница**. Она отвечает на вопрос, кто до чего вообще может достучаться.

На этой стадии инфраструктурной границы уже достаточно, чтобы удерживать тенанты друг от друга. `tenant-alpha` не должен ходить в `tenant-beta`. Запрос в контексте `tenant-alpha` не должен возвращать строки `tenant-beta`. Агент одного тенанта не должен быть вызван как инструмент из другого.

Но этого недостаточно, чтобы:

- фиксировать, кто именно инициировал действие.
- выражать тонкую настройку политик.
- строить полноценный audit.

Именно здесь границы заканчиваются, а Identity становится отдельным архитектурным слоем.

## Шесть рабочих границ

Полные Helm-шаблоны и SQL-миграции — в репозитории. Ниже только концептуальные иллюстрации, достаточно, чтобы понять форму каждой границы.

Кратко:

- **Namespace на тенант** — грубая, но главная runtime-граница.
- **NetworkPolicy deny-all + точечный allow** — enforced-изоляция сети.
- **PostgreSQL RLS** — data-изоляция на уровне БД.
- **TenantDB-обёртка** — RLS не работает без явной транзакции и `SET LOCAL`.
- **kagent `allowedNamespaces.from: Same`** — cross-tenant deny на уровне agent loop.
- **Пер-тенантные секреты** — токены мимо Helm values и истории релизов.

### 1. Один namespace = один тенант

Это самая грубая, но самая важная граница. Агенты не делят pod между отделами или организациями, а tenant namespace маркируется label'ами, которые затем используются в NetworkPolicy.

Если мультитенантность не выражена на уровне разграничения сред выполения(runtime boundary), дальше всё превращается в надежду на дисциплину кода.

### 2. NetworkPolicy

В каждом tenant namespace по умолчанию стоит `deny-all` для ingress и egress, а дальше открываются только нужные направления:

- `platform` — Postgres, GoTrue.
- `kagent` — A2A controller.
- `shared-tools` — общий MCP.
- DNS.
- внешний трафик на 80/443 для LLM-провайдеров.
- ingress из `kagent`.

Для `shared-tools` тоже действует ограничение: его могут вызывать только tenant namespace и `kagent`.

Это важная деталь, потому что изоляция здесь не декларативная, а реально enforced.

Полный набор политик — в чарте: `helm/charts/tenant/templates/networkpolicy.yaml`.

### 3. PostgreSQL RLS

На уровне данных изоляция реализуется через Row-Level Security.

Для tenant-таблиц используется политика вида:

```sql
USING (tenant_id = current_setting('app.tenant_id', true))
```

Роль приложения не должна обходить RLS. Если контекст не выставлен, база должна вернуть ноль строк, а не утечку.

Так data isolation перестаёт зависеть только от приложения.

**Trade-off: отдельная БД на тенант vs shared + RLS.** Идеальная изоляция — отдельная база (или кластер) на тенант: ошибся в RLS — утечки нет, потому что RLS нет. Цена: N баз = N бэкапов, миграций, коннект-пулов; операционная сложность растёт линейно с числом тенантов. Для mid-market и dev/sandbox это оверкилл. Этот референс выбирает компромисс: одна платформенная БД + RLS + TenantDB + fail-safe (`ALTER DATABASE ... SET app.tenant_id = ''`). Изоляция переносится с инфраструктуры на корректность RLS-модели — а значит, требует тестов на утечку и дисциплины обёртки (прямые `asyncpg`-вызовы запрещены). Меньше операционной сложности, больше ответственности за RLS. Для regulated/finance/HIPAA-класса компромисс пересматривается в сторону отдельной БД или кластера.

### 4. TenantDB как обязательная обёртка

RLS не спасает, если приложение работает с БД неправильно.

Здесь есть пара типовых граблей:

- `SET LOCAL` живёт только внутри транзакции.
- `asyncpg` легко ломает контекст, если не завернуть операции в явный `BEGIN … COMMIT`.

Поэтому TenantDB — это не удобная прослойка, а часть security model.

Обёртка принудительно держит явные транзакции и отключает кэш prepared statements, чтобы RLS не оставался “галочкой на бумаге”.

Суть обёртки — `SET LOCAL` всегда внутри явной транзакции:

```python
async with conn.transaction():
    await conn.execute(f'SET LOCAL "app.tenant_id" = \'{tenant_id}\'')
    return await conn.fetch(sql, *params)
```

Полный код и комментарий про `statement_cache_size=0` — в `packages/saas-common/saas_common/supabase_client.py`.

### 5. Запрет cross-tenant на уровне kagent

На уровне agent loop изоляция продолжается через `allowedNamespaces.from: Same` в `Agent` CRD.

Это значит, что агент можно вызвать как инструмент только из того же namespace.

Контроллер обеспечивает это правило на уровне системы агентов, а NetworkPolicy остаётся вторым уровнем защиты.

### 6. Пер-тенантные секреты

Секреты ботов сохраняются в namespace каждого тенанта во время деплоя.

Они не проходят через Helm values и не попадают в историю релизов.

При этом здесь всё ещё нет Vault и per-tenant path RBAC. Это следующий шаг, который логично появится вместе со слоем Identity.

## Слой Input

Входной слой здесь начинается не с абстрактного “пользователь прислал сообщение”, а с конкретного маршрута.

Сообщение приходит в tenant-specific Telegram-бота, затем пользователь резолвится через `platform.users`, после чего бот отправляет A2A `message/send` в `kagent-controller`.

Одна важная деталь здесь особенно характерна для реального запуска: `contextId` должен лежать внутри `params.message`, а не снаружи. Иначе kagent создаёт новый thread на каждом ходе.

## Слой Agent Loop

Agent Loop здесь построен на kagent и остаётся декларативным.

Никакого кастомного runtime-кода агента, никакого LangGraph, никакого `interrupt()`. Агент описывается как CRD, где задаются:

- `systemMessage`.
- `modelConfig`.
- список инструментов.
- `a2aConfig`.

На тенант поднимаются два агента:

- `echo-agent`.
- `management-agent`.

Второй вызывает первого как инструмент.

Это и есть правильный смысл agent-as-tool: агент — не чат-сессия, а управляемая структура вызовов

## Слой Execution

Execution вынесен в отдельные FastMCP-серверы, подключённые через `RemoteMCPServer` CRD.

Здесь важно разделять:

- `ping-tool` - общий, stateless.
- `tenant-info-tool` и `order-tracker` - привязанны к конкретному тенанту

Они поднимаются по одному экземпляру на namespace, получают `TENANT_ID` из окружения и работают с данными только через TenantDB.

Это принципиально важно: tenant identity для инструмента не приходит от вызывающей стороны как параметр, который можно подменить. Она задаётся deployment boundary.

## Развёртывание на Kind

Локальный запуск тоже показан как часть архитектуры, а не как вспомогательная деталь.

Здесь есть несколько типичных мест, где local Kubernetes setup ломается:

- локальный registry нужно зеркалировать в containerd на нодах Kind.
- `kind load docker-image` обязателен, иначе pod может продолжить использовать старый image.
- kagent работает с внешним Postgres, чтобы память агента и tenant data жили в одной платформенной базе, но под разными ролями.
- пересборка image завязана на хэш исходников.
- онбординг тенанта реализован как код через `tenant.py deploy`.

## Один запрос через весь стек

Самая важная проверка — не список компонентов, а один живой запрос.

Маршрут выглядит так:

1. Сообщение приходит в Telegram.
2. Бот резолвит пользователя.
3. Бот отправляет A2A `message/send`.
4. kagent поднимает сессию для декларативного агента.
5. Агент вызывает MCP-инструмент.
6. Инструмент идёт в TenantDB.
7. TenantDB выставляет `app.tenant_id`.
8. PostgreSQL возвращает только tenant-scoped данные.

На каждом шаге работают одни и те же границы.


## Где границы перестают работать

Границы решают не всё.

Они разделяют тенанты друг от друга, но не различают роли внутри тенанта.

Они не умеют:

- отличать пользователей.
- фиксировать субъекта действия.
- выражать policy уровня пользователя.
- строить полноценный audit.
- переживать компрометацию pod внутри tenant namespace.

Именно поэтому Identity, Policy & Audit это необходимая часть архитектуры.

## Что пока не покрыто и чем это опасно

Границы разделяют тенантов, но внутри tenant namespace они открыты. Угрозы, которые остаются:

- **Компрометация pod внутри tenant namespace.** Попав в один под `tenant-alpha`, атакующий проходит через все границы между ним и данными `tenant-alpha`: RLS открывает строки, NetworkPolicy пускает внутри ns, секреты ns доступны. Borders — контроль между тенантами, не внутри.
- **Привилегированные контейнеры и hostPath.** `privileged: true` или `hostPath`-mount выводят за NetworkPolicy и namespace; изоляция на уровне K8s перестаёт работать. В этом референсе нет PodSecurityPolicy/admission, ограничивающего это.
- **Общий `shared-tools` как общая зона поражения.** `ping-tool` один на все тенанты. Уязвимость в общем MCP-сервере или его зависимостях — точка отказа/компрометации для всех тенантов сразу. База shared-инструментов должна быть минимальной и stateless.
- **RLS тихо не работает.** Забытый `SET LOCAL` или кэш prepared statements asyncpg — и запрос возвращает не те строки или 0 строк, без видимой ошибки. Нет автоматического теста на утечку — только проверкой.
- **Нет audit trail.** Кто инициировал действие внутри тенанта — не фиксируется. `platform.audit_log` есть, но в путь запроса не пишется.

Когда Borders-без-Identity становится недостаточно и слой Identity обязателен:

- регуляторные требования: нужен audit и контроль уровня субъекта.
- несколько ролей внутри тенанта: borders их не различают.
- привилегированные действия (изменение политики, доступ к чужим данным, исполнение кода): требуют approval и записи.
- shared-инструменты с состоянием или чувствительным доступом: требуют per-tenant policy, не общего ingress-разрешения.

## Зачем всё это

Если конечная цель — один ReAct-агент, то вся эта архитектура выглядит чрезмерной. Namespace на тенант, NetworkPolicy, RLS, TenantDB, kagent-изоляция, секреты мимо Helm — много инфраструктуры ради одного цикла «подумал → ответил».

Но для enterprise AI один цикл обычно становится лишь первым шагом. По мере усложнения задач появляются дополнительные проверки, специализированные роли, верификация результатов, исправление ошибок, тестирование и повторные попытки. Сначала всё это обычно остаётся внутри Agent Loop: агент вызывает другие агенты как инструменты, запускает проверки, повторяет шаги, следит за бюджетом и принимает решение, когда остановиться.

На определённом этапе возникает новая проблема. Сам loop продолжает работать, однако процесс становится слишком сложным, чтобы оставаться скрытым внутри одного цикла. Его уже недостаточно просто исполнять — его нужно проектировать, наблюдать и отлаживать.

Именно здесь появляется Graph Engineering.

Graph engineering не заменяет Loop Engineering и не делает того, чего loop принципиально не умеет. Но он добавляет новый уровень абстракции, где объектом проектирования становится уже не цикл агента, а сам процесс выполнения задачи.

```
fan-out → judge → verifier → fixer → tests → pass / retry
```

В графе инженер управляет уже не только поведением агента, но и структурой самого процесса. Такой граф невозможен без доверенных границ. Каждый узел должен понимать, в каком тенанте он работает, кто инициировал действие и по каким правилам может вызвать следующий узел. Без RLS узлы перетекают друг в друга. Без NetworkPolicy один скомпрометированный под становится точкой отказа для всех. Без Identity нет субъекта действия, а значит, audit и approval невозможны. Без TenantDB граница данных превращается в надежду на дисциплину кода.

Именно поэтому эта серия не пытается рассказать про Enterprise AI Harness за одну статью. Enterprise AI Harness - это не отдельная технология и не один фреймворк, а архитектурная модель. Поэтому каждый материал разбирает один из её аспектов: границы, изоляцию, доверенную идентичность, инженерные подходы и компоненты, которые вместе превращают набор AI-инструментов в полноцененную корпоративную платформу.

Не `LLM → Agent → Tools → MCP`, а `Architecture → Isolation → Identity → Graph`. Graph engineering здесь не «следующая тема», а ответ на вопрос, ради чего строилась вся инфраструктура: не для того, чтобы запустить Telegram-бота, а для того, чтобы позже можно было безопасно оркестрировать десятки агентов, каждый из которых знает свой тенант, своего инициатора и свои правила.

Без Identity и Audit такой граф быстро упирается в потолок: состояние сохранять можно, skills и prompts переписывать можно, multi-agent judging запускать можно — но без доверенной идентичности и следа действий система остаётся инженерно интересной, но enterprise-незавершённой. Поэтому порядок жёсткий: сначала границы, потом Identity, потом граф поверх. Иначе получится красивый граф, у которого есть узлы, рёбра и амбиции, но нет доверия.
